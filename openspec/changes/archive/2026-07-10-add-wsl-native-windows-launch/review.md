## Findings

### ✅ Verified - RF1 The real WSL launcher path (`cmd.exe /c start`) will mangle the launch URL because `&` is a cmd metacharacter
- **Severity:** High
- **Evidence:** `apps/desktop/scripts/livedocs-wsl.mjs:39` runs `spawnSync('cmd.exe', ['/c', 'start', 'LiveDocs', url], ...)`. The URL always contains an unquoted `&` (`livedocs://wsl/open?distro=...&path=...`; see the smoke output `...?distro=Ubuntu&path=%2F...`). WSL interop assembles the Windows command line by joining args without quoting, so `cmd.exe` parses `&` as a command separator: the app receives a URL missing the `path` param (`parseWslLaunchUrl` returns `null`, request silently dropped) and cmd attempts to execute the trailing `path=...` token as a command. The smoke check (`smoke-wsl-native.mjs`) always overrides `LIVEDOCS_WINDOWS_LAUNCHER`, so the default `cmd.exe` path is never exercised; task 6.4 (real-Windows verification) is the one unchecked task, which is exactly where this would surface.
- **Recommendation:** Avoid raw `cmd.exe start` argument passing. Either pass a single pre-quoted command string (`spawnSync('cmd.exe', ['/c', `start "LiveDocs" "${url}"`])`), escape `&` as `^&`, or use `powershell.exe -Command Start-Process '<url>'` / `rundll32 url.dll,FileProtocolHandler <url>`. Add a smoke/unit assertion that the launch command survives URLs containing `&`.
- **Fix:** `livedocs-wsl.mjs` now builds the default launch as a single quoted `cmd.exe /d /s /c start "" "<url>"` command, and `smoke-wsl-native.mjs` dry-runs the default command to assert the quoted URL still contains `&path=`.
- **Verification:** Confirmed `defaultWindowsLaunchCommand` passes the whole `start "" "<url>"` string as one argument with the URL double-quoted (and embedded quotes doubled), which makes `&` literal under cmd parsing; the smoke dry-run assertion catches regressions. `pnpm smoke:wsl-native` passes. Final confirmation on real Windows interop still lands with task 6.4, which remains open.

### ✅ Verified - RF2 The WSL agent never exits when its stdin closes, leaving stale agent processes in WSL
- **Severity:** High
- **Evidence:** `apps/desktop/src/main/wsl-agent-runner.ts:62-79` reads stdin and handles SIGTERM/SIGINT, but registers no `process.stdin.on('end' | 'close')` handler. `WslAgentClient.dispose()` sends `agent.shutdown` with a 1s grace then kills `wsl.exe` (`agent-client.ts:89-103`), but the Linux-side agent process is not in the Windows process tree — if the Electron app crashes, is force-killed, or the `wsl.exe` relay dies, the agent (with active chokidar watchers and an open SQLite store) lingers indefinitely. design.md explicitly calls this out as a risk to mitigate ("Agent lifecycle bugs could leave stale WSL processes running -> ... add heartbeat/shutdown handling"). The e2e teardown (`scenarios.spec.ts` afterAll `proc.kill()`) also orphans the agent in the test environment.
- **Recommendation:** In `runWslAgent`, treat stdin `end`/`close` as a shutdown signal (`process.stdin.on('end', () => void service.close().finally(() => process.exit(0)))`). Optionally add a heartbeat timeout as defense in depth.
- **Fix:** `runWslAgent` now uses a single shutdown path for stdin `end`/`close`, SIGTERM, and SIGINT, closing `NodeWorkspaceService` before exiting.
- **Verification:** `wsl-agent-runner.ts:62-84` registers an idempotent `shutdown` handler on stdin `end`, stdin `close`, SIGTERM, and SIGINT that closes the service (watchers, SQLite) then exits 0. An orphaned agent now exits when the pipe drops, including the e2e `proc.kill()` teardown path.

### ✅ Verified - RF3 Handshake failure leaves the spawned agent process running and the client in a reusable-but-broken state
- **Severity:** Medium
- **Evidence:** `apps/desktop/src/main/agent-client.ts:54-67` — when `isProtocolVersionCompatible` fails, `connect()` throws but never kills `this.child`. The incompatible agent process stays alive (compounding RF2), and `this.child` remains set, so a later `request()` via `ensureChild()` would happily talk to the incompatible agent, bypassing the handshake guard. The spec requires the open to "fail with a clear compatibility error" — the error is produced, but the process/state cleanup is missing.
- **Recommendation:** On handshake failure (and on any `connect()` error), call `dispose()`/kill the child and reset `this.child = null` before rethrowing.
- **Fix:** `WslAgentClient.connect()` now stops and clears the child on handshake/setup errors, and compatibility failures are rethrown as `Error` instances with the protocol code attached.
- **Verification:** `connect()` wraps the handshake in try/catch and calls `stopChild(err)` (kills the child, destroys pipes, clears `this.child`, fails pending requests) before rethrowing; the `stoppingChild` bookkeeping correctly suppresses the spurious `onExit` disconnect event for deliberately stopped children (`agent-client.ts:52-83`, `219-232`).

### ✅ Verified - RF4 Launch-bridge open requests can produce unhandled promise rejections in the main process
- **Severity:** Medium
- **Evidence:** `openWorkspaceRequest` rethrows after broadcasting (`workspace-router.ts:46-52`). Callers in `launch-bridge.ts:15-23` (`void openFromArgv(argv)`, `void openLaunchUrl(url)`) and `index.ts:134` (`await openFromArgv(process.argv)` inside `app.whenReady().then(...)` with no catch) leave those rejections unhandled. A `livedocs://` request for a nonexistent path or a failed agent spawn (e.g. distro not installed) triggers an unhandled rejection in the Electron main process (fatal under Node's default `--unhandled-rejections=throw`). Note `index.ts:137-143` wraps the `LIVEDOCS_WORKSPACE` preopen in try/catch — the launch-bridge paths deserve the same care.
- **Recommendation:** Catch and log errors inside `openFromArgv`/`openLaunchUrl` (the renderer is already notified via the `workspace:connection` error broadcast), or `.catch()` at each call site.
- **Fix:** `openLaunchUrl` now catches failed workspace opens, logs the failure, and returns `false` after the router broadcasts the connection error.
- **Fix:** `parseWslLaunchUrl` now catches invalid WSL workspace reference fields and returns `null`, making `findWslLaunchUrl` total for malformed `livedocs://wsl/open` URLs. `workspace-ref.test.ts` covers a malformed relative-path launch URL and confirms a later valid URL is still found.
- **Verification (2026-07-09):** Confirmed complete. `parseWslLaunchUrl` ([wsl-launch.ts:5-27](packages/store/src/wsl-launch.ts#L5-L27)) now wraps both of its throw sources — `new URL()` and `createWslWorkspaceReference()` — in try/catch, so it is total; `findWslLaunchUrl` (which only calls it) therefore can no longer throw out of the `second-instance` handler at `launch-bridge.ts:28`, and `openLaunchUrl` still catches `openWorkspaceRequest` failures for well-formed-but-failing opens. Both halves of the original finding are covered. Regression test ([workspace-ref.test.ts:70-77](packages/store/test/workspace-ref.test.ts#L70-L77)) asserts the malformed relative-path URL yields `null`, `findWslLaunchUrl` doesn't throw, and a later valid URL is still found. `pnpm test` (57/57) and `@livedocs/store` typecheck pass.

### ✅ Verified - RF5 A failed WSL open silently closes the previously active workspace without telling the renderer
- **Severity:** Medium
- **Evidence:** `workspace-router.ts:37-53` closes the current backend *before* attempting the new open. If the agent open then fails, the old session is gone (`session.closeWorkspace()` broadcasts nothing), `currentBackend().current()` returns `null`, but the renderer still displays the old workspace — every subsequent IPC call throws "No workspace is open". Task 3.5 and the spec promise surfacing agent errors "without losing the existing local workspace experience".
- **Recommendation:** Either open the new backend first and only close the previous one on success, or broadcast `workspace:changed: null` after closing so the renderer state is consistent with the main process.
- **Fix:** `workspace-router` now opens the next backend before closing the previous backend, and `AgentWorkspaceBackend.open()` stages replacement WSL clients until `workspace.open` succeeds while disposing failed pending clients.
- **Verification:** Correct. The router opens the next backend first and only closes/switches on success (`workspace-router.ts:37-55`); the agent backend stages the new client, restores `client`/`info` and disposes the staged client on failure, and the `this.client !== client` guards suppress stale-client events (`agent-backend.ts:32-88`). The local session path got the same open-before-close treatment (`session.ts` `openWorkspace`). Note: events the agent emits *during* a pending open (initial `index.status` scanning) are suppressed by the guard, but the renderer refetches index status via `refreshWorkspaceState` after `workspace:changed`, so nothing is lost.

### ✅ Verified - RF6 GitService now reports "not a repo" for any workspace opened at a subdirectory of a repository — an unspecced local behavior change
- **Severity:** Medium
- **Evidence:** `packages/analysis/src/git.ts:82-88` adds `isWorkspaceRootRepo()`, which requires `--show-toplevel` to equal the workspace root, and gates `info()`, `recentCommits()`, `fileHistory()`, and `recentDiff()` on it. Previously, opening a subfolder of a monorepo showed branch/commit data; now Git features are disabled entirely for that case (in both the local and WSL paths). No spec/design/task calls for this, and tasks 1.3/7.2 promise unchanged local behavior. The new test (`analysis.test.ts` "does not treat a child folder as the opened repository root") shows it is deliberate at the code level, but the artifacts don't record the decision.
- **Recommendation:** If intentional (avoiding parent-repo commits leaking into a child workspace), record it in design.md/spec deltas and consider scoping `git log` to the subdirectory instead of disabling Git entirely. If not intentional, restore the previous `checkIsRepo` behavior.
- **Fix:** `GitService` now treats worktree subdirectories as Git-enabled workspaces, scopes recent commits/file history/recent diffs with top-relative Git pathspecs, translates commit file lists back to workspace-relative paths, and records the intended behavior in design/spec artifacts.
- **Verification:** Correct and well-covered. `repoContext()` yields a `:(top)`-prefixed pathspec for subdir workspaces and none at the repo root (unchanged root behavior); commit file paths are relativized with out-of-scope files filtered. Empirically confirmed `git log --follow -- ':(top)sub/file'` works. The decision is now recorded in design.md ("Scope Git integration to the opened workspace inside larger worktrees") and as a spec scenario ("WSL Git worktree subdirectory"), and the new analysis test covers branch info, scoped commits, scoped file history, and scoped diffs.

### ✅ Verified - RF8 Workspace IO logic is duplicated between the local session path and NodeWorkspaceService
- **Severity:** Medium
- **Evidence:** `local-backend.ts:47-87` and `node-workspace-service.ts:105-139` contain byte-identical `readFile`/`applyEdit` implementations; the watcher→index→staleness→git-refresh pipeline is likewise implemented twice (`session.ts:78-102` vs `node-workspace-service.ts:224-255`), as is `gitOverview`. design.md's migration plan step 2 intended one service entrypoint that "can run in-process for local workspaces and out-of-process in WSL"; the local path never adopted `NodeWorkspaceService`, so future fixes (e.g. to the applyEdit ambiguity rules) must be made in two places.
- **Recommendation:** Either route `LocalWorkspaceBackend` through `NodeWorkspaceService` (keeping the worker-thread indexer as a constructor strategy), or at minimum extract shared `readFile`/`applyEdit` helpers next to `resolveInWorkspace` in workspace-paths.ts.
- **Fix:** `NodeWorkspaceService` now extends a shared `WorkspaceService` that owns tree/file read/apply edit/search/Git overview/generated artifact/index watcher behavior, while the local session injects a worker-thread indexer driver, Electron event broadcasts, and local AI provider access. `LocalWorkspaceBackend` now delegates those operations to the shared service instead of duplicating them.
- **Verification:** Genuine consolidation, not a cosmetic wrapper. `readFile`/`applyEdit`/`gitOverview`/watch→index→staleness→git-refresh now exist once in `WorkspaceService` (node-workspace-service.ts); the local path composes it via `createLocalService()` with a `WorkerIndexerDriver` (worker-thread `IndexerHost`), Electron broadcasts, and an AI-capable `GeneratorRuntime`, while the agent uses the direct in-process indexer and no AI. `generator-host.ts` was decoupled from the Electron `Session`/app-store, so `gen.get`/`gen.refresh` behavior (including the ai-unconfigured path) is shared too. Local Git + WSL agent e2e scenarios pass against the refactor.

### ✅ Verified - RF10 Launcher helpers in @livedocs/store are dead code duplicated by the standalone launcher script

### ✅ Verified - RF11 Agent protocol requests have no timeout
- **Severity:** Low
- **Evidence:** `agent-client.ts:70-87` stores pending promises that only settle on a response line, process error, or exit. A wedged agent (e.g. blocked on a hung filesystem/network mount inside WSL) leaves renderer invocations pending forever with no feedback beyond the initial `connecting` event.
- **Recommendation:** Add a configurable per-request timeout (generous for `workspace.open`/indexing-adjacent calls) that rejects with a protocol error so the UI can surface it.
- **Fix:** `WslAgentClient` now applies a configurable per-request timeout with a 60s default, clears timeout handles on response/failure, and rejects timed-out requests with a surfaced workspace error.
- **Verification:** Correct. Timers are created per request (skippable via `requestTimeoutMs <= 0`), cleared on response, on `failAll`, and on synchronous write failure; timed-out requests reject with a coded error the renderer surfaces. No leaked-timer path found.

### ✅ Verified - RF12 `deserializeWorkspaceReference` accepts a non-string `name`, crashing later instead of failing validation
- **Severity:** Low
- **Evidence:** `workspace-ref.ts:125-144` validates `kind`/`path`/`distro` types but passes `candidate.name` through unchecked; a numeric/object `name` (e.g. from a hand-crafted `--workspace` argument or malformed recents row) throws `TypeError: name.trim is not a function` inside `optionalName` rather than the intended "Invalid workspace reference" error.
- **Recommendation:** Coerce or validate: `typeof candidate.name === 'string' ? candidate.name : undefined`.
- **Fix:** `deserializeWorkspaceReference` now validates `name` before normalization, and `workspace-ref.test.ts` covers non-string serialized names.
- **Verification:** `deserializeName` rejects non-string/undefined names with "Invalid workspace reference" before any `.trim()` call, and the new test asserts the thrown message for a numeric name. Passes in the suite.
