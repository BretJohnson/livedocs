## Findings

### ✅ Verified - RF1 A transient temp file disappearing during a watch flush terminates the whole dev session
- **Severity:** Medium
- **Evidence:** In dev mode every debounced watcher flush calls `reconcileSource` → `collectSourceFiles`, which walks the entire source tree and reads each file with `hashFile` (`readFileSync`) and `statSync` with no error handling ([wsl-windows-core.mjs:106-125](../../../apps/desktop/scripts/wsl-windows-core.mjs#L106-L125)). Editors routinely create and delete short-lived temp/swap files (e.g. `4913`, `.goutputstream-*`, atomic-save temporaries). If such a file is enumerated by `readdirSync` but vanishes before `statSync`/`readFileSync`, an `ENOENT` propagates out of `flush()`, which calls `rejectFailure(...)` ([wsl-windows.mjs:131-138](../../../apps/desktop/scripts/wsl-windows.mjs#L131-L138)); that rejection wins the `Promise.race` in `runWindowsHelper` ([wsl-windows.mjs:230](../../../apps/desktop/scripts/wsl-windows.mjs#L230)) and tears down the session. `chokidar`'s `awaitWriteFinish` gates event *emission*, not the independent full-tree walk, so it does not protect against this. The result is spurious "synchronization failed" session kills during ordinary editing — the opposite of the design intent that only *unrecoverable* sync errors should terminate.
- **Recommendation:** Treat `ENOENT` (and other transient races) from `statSync`/`hashFile`/`cpSync` inside `collectSourceFiles`/`reconcileSource` as "file no longer present, skip it" rather than fatal, and reserve session termination for genuinely unrecoverable errors (permission failures on the mirror, watcher `error` events).
- **Fix:** Source collection, directory traversal, and copying now skip transient `ENOENT`/`ENOTDIR` races while preserving fatal handling for other IO errors. Deterministic tests cover both collection and copy disappearance.
- **Verification:** ✅ Confirmed. `collectSourceFiles` guards the child `readdirSync` (skips `ENOENT`/`ENOTDIR` only for non-root prefixes) and the per-file `readDetails` in try/catch, re-throwing non-transient errors ([wsl-windows-core.mjs:110-149](../../../apps/desktop/scripts/wsl-windows-core.mjs#L110-L149)); `reconcileSource` wraps the copy and drops the vanished file from the manifest on a transient race ([wsl-windows-core.mjs:226-236](../../../apps/desktop/scripts/wsl-windows-core.mjs#L226-L236)). The `ignores transient source disappearance during collection and copy` test asserts a vanished file yields `size 0` / `copied 0` rather than throwing, and the full suite passes (18/18 in the two WSL-windows files).

### ✅ Verified - RF2 Ctrl+C / termination cleanup uses `taskkill /PID <child.pid>`, but the WSL-interop child pid may not be a valid Windows PID
- **Severity:** Medium
- **Evidence:** The orchestrator spawns Windows Node through the `/mnt/...` interop path and, on SIGINT/SIGTERM or watcher failure, runs `spawnSync('taskkill.exe', ['/PID', String(child.pid), '/T', '/F'])` ([wsl-windows.mjs:213-221](../../../apps/desktop/scripts/wsl-windows.mjs#L213-L221)). For a Windows process launched via WSL interop, `child.pid` is the Linux-side relay pid, not the Windows PID that `taskkill` operates on, so this call can target the wrong process or none at all. The design explicitly delegates descendant ownership to the Windows helper "so it can close them as a group," but the helper only reaps its group if it actually receives a termination signal — which depends on this teardown working. This is precisely the design risk *"Ctrl+C does not terminate a Windows descendant,"* and its verification (tasks 4.5 and 7.4) is still open, so the path is unproven. `verification.md` did not exercise interruption because the real Windows install did not complete.
- **Recommendation:** Verify end-to-end that Ctrl+C and unexpected child exit leave no orphaned Electron/Vite/agent processes (task 7.4). If `taskkill /PID` proves unreliable across interop, kill the Windows tree from *inside* the helper on a signal/heartbeat it can observe (e.g. parent-pipe close), or resolve the real Windows PID before calling `taskkill`. Add the process-level test with a fake helper (task 4.5) so this is regression-protected.
- **Fix:** The WSL orchestrator now closes a dedicated control pipe instead of passing its Linux relay PID to `taskkill`. The Windows helper observes pipe closure and terminates the tree by its real Windows PID; it also discovers and removes descendants after an unexpected direct-child exit. Fake-helper tests and a real Ubuntu-to-Windows Node smoke cover interrupt, unexpected exit, status propagation, and descendant cleanup.
- **Verification:** ✅ Confirmed. `runManagedInterop` spawns with `stdio: ['pipe',...]` and, on signal/watcher-failure, ends the child's stdin and arms a `SIGTERM` fallback timer ([wsl-windows-process.mjs:54-116](../../../apps/desktop/scripts/wsl-windows-process.mjs#L54-L116)); the helper resumes stdin and, on `end`/`close`, runs `taskkill /PID <its own child pid>` — a genuine Windows PID because the helper runs on Windows Node — and sweeps orphaned descendants via a `Win32_Process` walk after any child exit ([wsl-windows-helper.mjs:14-45](../../../apps/desktop/scripts/wsl-windows-helper.mjs#L14-L45),[113-143](../../../apps/desktop/scripts/wsl-windows-helper.mjs#L113-L143)). I ran the process-lifecycle tests: `closes the helper control pipe on interruption and reports 130` spawns a real helper + real descendant, emits SIGINT, and asserts the descendant is dead and exit code is 130; `stops the helper before propagating a watcher failure` confirms the helper is stopped before the failure surfaces. The pnpm-relay PID is no longer used for teardown. The real WSL-relay leg is exercised only by the opt-in `smoke:wsl-windows:integration` (env-gated; recorded as passing in `verification.md`) — the mechanism itself is verified here by the passing lifecycle tests.

### ✅ Verified - RF3 Every incremental watch flush re-hashes the entire source tree and re-hashes every unchanged mirror file
- **Severity:** Low
- **Evidence:** `collectSourceFiles` hashes all source files on every call ([wsl-windows-core.mjs:110-125](../../../apps/desktop/scripts/wsl-windows-core.mjs#L110-L125)), and `reconcileSource`'s copy predicate additionally calls `hashFile(destination)` for every file whose manifest hash matched (`previous hash === current` → `existsSync` → `hashFile(destination)`) ([wsl-windows-core.mjs:191-202](../../../apps/desktop/scripts/wsl-windows-core.mjs#L191-L202)). So a single one-file save in a large monorepo drives O(entire-repo) reads on both the WSL source and the `/mnt/c` mirror per debounced flush — the `/mnt/c` reads being the slow path the design already flags. The full-source rehash is the dominant cost; the destination rehash exists only to detect manual mirror edits, which the design already declares unsupported/disposable.
- **Recommendation:** For incremental flushes, use `mtime`/`size` as the fast change signal (falling back to content hash only on ambiguity), and drop or gate the per-file destination re-hash. The initial reconciliation can keep the stronger check. This is a performance/robustness improvement, not a correctness bug.
- **Fix:** Source manifests now record `mtimeMs` and size so incremental reconciliation can reuse compatible hashes, and incremental flushes no longer hash every destination. Initial reconciliation retains destination content verification and repairs unsupported mirror edits.
- **Verification:** ✅ Confirmed. `collectSourceFiles` reuses the prior manifest entry when `size` and `mtimeMs` match ([wsl-windows-core.mjs:114-124](../../../apps/desktop/scripts/wsl-windows-core.mjs#L114-L124)), and the destination re-hash is now gated behind `!options.incremental` ([wsl-windows-core.mjs:224](../../../apps/desktop/scripts/wsl-windows-core.mjs#L224)); the dev watcher calls `reconcileSource(..., { incremental: true })` ([wsl-windows.mjs:134](../../../apps/desktop/scripts/wsl-windows.mjs#L134)). The `uses fast metadata reconciliation incrementally and full verification initially` test proves an unsupported mirror edit is ignored incrementally (`copied 0`) but repaired on a full reconcile (`copied 1`), so the strong check is preserved for the initial pass. Efficiency improvement with no correctness regression.

### ✅ Verified - RF5 Watcher ignore list and `shouldSync` exclusion policy are duplicated and can drift
- **Severity:** Low
- **Evidence:** `isWatchCandidate` hardcodes its own exclusion list ([wsl-windows.mjs:160-176](../../../apps/desktop/scripts/wsl-windows.mjs#L160-L176)) that is a strict subset of the authoritative policy in `shouldSync`/`EXCLUDED_DIRECTORY_NAMES` ([wsl-windows-core.mjs:23-104](../../../apps/desktop/scripts/wsl-windows-core.mjs#L23-L104)) — it omits `.hg`/`.svn` and the `.env`/`.log`/`.DS_Store` file rules. Consequently the watcher schedules full reconciles for changes (e.g. `.env` writes, `*.log` churn) that `shouldSync` then filters out, so the result is correct but wasteful, and the two lists must be kept in sync manually. A future exclusion added only to `EXCLUDED_DIRECTORY_NAMES` would silently keep triggering flushes.
- **Recommendation:** Derive the watcher's ignore predicate from the shared `shouldSync`/exclusion set in `wsl-windows-core.mjs` so there is one source of truth.
- **Fix:** The watcher now delegates its candidate predicate directly to exported `shouldSync`; tests cover repository metadata, environment, log, and platform metadata exclusions.
- **Verification:** ✅ Confirmed. `isWatchCandidate` is now a one-line delegation to `shouldSync(relative, false)` ([wsl-windows.mjs:169-171](../../../apps/desktop/scripts/wsl-windows.mjs#L169-L171)), removing the duplicated list, so `EXCLUDED_DIRECTORY_NAMES` is the single source of truth. The `uses one exclusion policy for source synchronization and watching` test asserts `.hg`, `.svn`, `.env`, `*.log`, and `.DS_Store` are all excluded while `.env.example` is kept.

### ✅ Verified - RF6 Production/installer artifact location is left only inside the disposable mirror without a confirmed policy decision
- **Severity:** Low
- **Evidence:** For `build` and `dist`, the helper leaves output in the mirror and only reports the Windows path (`session.artifacts.build` / `session.artifacts.dist`) to the terminal ([wsl-windows-helper.mjs:47-52](../../../apps/desktop/scripts/wsl-windows-helper.mjs#L49-L52)), i.e. under `%LOCALAPPDATA%\LiveDocs\dev-mirrors\<id>\apps\desktop\{out,release}`. Task 5.3 called for an explicit *decision* on whether artifacts stay in the mirror or are copied to a documented WSL-visible output location, and the design's open question raises the same point. The current behavior places real installer output inside a directory the change itself marks `DISPOSABLE: ... Do not edit` and that `clean` will delete — so a `dist` artifact a developer wants to keep lives only in a location that cleanup destroys, and it is never copied anywhere WSL-visible for distribution.
- **Recommendation:** Confirm and document the intended policy. If artifacts are meant to persist, copy completed installers to a documented WSL-visible output path (or exempt them from `clean`); if staying in the mirror is intentional, state that explicitly in the docs and note the interaction with `clean:windows-from-wsl`.
- **Fix:** The developer documentation now explicitly defines mirror-local artifacts as the intentional cache-oriented policy and warns that `clean` removes build and installer output, so installers must be copied elsewhere before cleanup when retention is required.
- **Verification:** ✅ Confirmed. `docs/wsl-native-windows.md` now states artifacts stay in the mirror as an "intentional cache-oriented policy," instructs copying installers elsewhere before cleanup if they must be retained, and documents that `clean` deletes all build and installer artifacts inside the ownership-validated mirror (docs/wsl-native-windows.md:41-44). Task 5.3's decision is now explicit and matches the implementation.

### ✅ Verified - RF7 The cross-interop termination behavior is an unresolved verification obligation, not just a code concern
- **Severity:** Medium
- **Evidence:** Related to RF2 but tracked separately as the verification gap it creates: it is currently unknown whether `taskkill /PID <interop-child-pid> /T /F` ([wsl-windows.mjs:221](../../../apps/desktop/scripts/wsl-windows.mjs#L221)) actually tears down the Electron/Vite tree, or whether cleanup only works incidentally when the WSL relay dies on Ctrl+C. Because task 7.4 is unchecked and `verification.md` never reached a running Windows UI, there is no evidence either way, so the design's "make process ownership explicit" guarantee is unverified.
- **Recommendation:** Resolve empirically as part of task 7.4 — start a real dev session, Ctrl+C it and separately force-kill the Windows child, and confirm via the WSL/Windows process lists that no session-owned agent, build, or Electron process survives. Record the result here so RF2/RF7 can be closed with evidence rather than assumption.
- **Fix:** The opt-in integration smoke now exercises the actual Ubuntu WSL interop relay and production Windows helper in lifecycle-only mode. It verifies both control-pipe interruption and unexpected Windows child exit against a fake build tree, queries Windows process state, and confirms neither the child nor descendant survives.
- **Verification:** ✅ Confirmed as an executable obligation. `smoke-wsl-windows-integration.mjs` gains `runLifecycleBoundarySmoke`, runnable in isolation via `LIVEDOCS_WSL_WINDOWS_LIFECYCLE_ONLY=1`: it stands up an owned mirror, launches the real helper across the interop relay, emits SIGINT, asserts exit code 130, and then queries `Get-Process` for the helper's parent+descendant PIDs asserting the count is 0; it repeats for an unexpected child exit (code 23) with the same descendant-survival check ([smoke-wsl-windows-integration.mjs:41-168](../../../apps/desktop/scripts/smoke-wsl-windows-integration.mjs#L41-L168)). This turns the previously open question into a repeatable real-boundary check. It is env-gated (WSL-only, opt-in) so I could not run it from this Windows-side session; `verification.md` records it passing on a real Ubuntu-to-Windows setup, and the platform-independent lifecycle tests I ran here confirm the underlying mechanism.
