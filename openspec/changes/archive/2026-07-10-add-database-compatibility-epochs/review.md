## Findings

### âś… Verified - RF1 An unreadable/corrupt existing database file throws before the reset logic and leaks the SQLite handle
- **Severity:** Medium
- **Evidence:** In [database-lifecycle.ts:42-43](packages/store/src/database-lifecycle.ts#L42-L43) the initial `new DatabaseConstructor(...)` and `readCompatibilityEpoch(db)` run *outside* the `try` block that begins at line 61. `readCompatibilityEpoch` issues a `SELECT` against `sqlite_master`; if the on-disk file exists with `size > 0` but is not a valid SQLite database (corruption, truncated/partial file from a crash, or a non-DB file), that query throws `SQLITE_NOTADB` and propagates out of `openCompatibleDatabase` without ever calling `db.close()`. Result: (a) the freshly opened connection is leaked, and (b) startup/workspace-open aborts with a raw SQLite error instead of the intended recovery path. This partially defeats the feature's stated goal of "reset legacy or epoch-incompatible databases before current code queries them" â€” a corrupt `app.db` bricks startup rather than being recreated.
- **Recommendation:** Wrap the initial open + `readCompatibilityEpoch` in a guard. Treat a read failure the same as a missing/legacy epoch (close the handle, then take the removal-and-recreate path), or at minimum `db.close()` and rethrow a focused error. Add a test seeding a garbage file at the target path.
- **Fix:** Guarded initial construction and epoch inspection, closes the inspection handle on read failure, and recreates unreadable databases through the normal reset path. Added a garbage-file regression test.
- **Verification:** Confirmed. [database-lifecycle.ts:43-62](packages/store/src/database-lifecycle.ts#L43-L62) now wraps the initial `new DatabaseConstructor` in try/catch and guards `readCompatibilityEpoch`; on a read failure it calls `db.close()`, nulls the handle, and sets `resetFrom = 'unreadable'`, which routes into the standard removal-and-recreate path (lines 70-82). The construction failure case also throws a focused "Failed to inspect ... database compatibility epoch" error rather than leaking. Regression test [store.test.ts:161-185](packages/store/test/store.test.ts#L161-L185) seeds `'not a sqlite database'`, then asserts recovery to an empty store and the `unreadable -> 1` reset diagnostic. Suite green (20/20), typecheck clean.

### âś… Verified - RF2 The future-version refusal path writes to the database before the guard runs, contrary to the "fails without modifying the database" requirement
- **Severity:** Low
- **Evidence:** The requirement *"Future migration versions are not silently destroyed"* states opening SHALL fail *"without deleting or modifying the database"* ([spec.md:51-56](openspec/changes/add-database-compatibility-epochs/specs/database-schema-lifecycle/spec.md#L51-L56)). In [database-lifecycle.ts:61-64](packages/store/src/database-lifecycle.ts#L61-L64), `initializeCompatibilityEpoch` (a `CREATE TABLE IF NOT EXISTS` + `INSERT OR REPLACE` write) and `configure` run *before* `runMigrations` performs the `current > latest` check that raises the "newer than supported" error. Because a matching epoch always already has the metadata row, this is a same-value rewrite (and the passing test in [store.test.ts:161-193](packages/store/test/store.test.ts#L161-L193) still observes `user_version`/data unchanged), so no user data is lost â€” but a redundant write transaction/WAL frame is emitted against a database the spec says should not be touched on this failure path.
- **Recommendation:** Validate migration direction before any write. E.g. read `user_version` and compare against the migration list max first (or move the future-version check ahead of `initializeCompatibilityEpoch`), so the failure occurs with zero writes.
- **Fix:** Extracted migration-version validation and invokes it before metadata initialization or database configuration. The future-version test now installs a trigger that fails any metadata write, proving refusal is read-only.
- **Verification:** Confirmed. `validateMigrationVersion` is now a standalone export ([migrations.ts:135-145](packages/store/src/migrations.ts#L135-L145)) and is called at [database-lifecycle.ts:89](packages/store/src/database-lifecycle.ts#L89) *before* `initializeCompatibilityEpoch` and `configure`. The refactored [store.test.ts:224-263](packages/store/test/store.test.ts#L224-L263) seeds a future `user_version = 2` and installs a `BEFORE INSERT ... RAISE(ABORT, 'metadata write attempted')` trigger on the metadata table; because the assertion is `/newer than supported/` (not the trigger message), the test proves the guard fires with zero writes â€” if the old ordering remained, the trigger would abort first. `runMigrations` still re-validates internally, which is harmless. Suite green (20/20).

### âś… Verified - RF3 No test exercises the "reset cannot complete" safe-failure requirement
- **Severity:** Low
- **Evidence:** The spec requires that when a file/companion cannot be removed or the fresh schema cannot be created, "store opening fails before application reads or writes are attempted" ([spec.md:47-49](openspec/changes/add-database-compatibility-epochs/specs/database-schema-lifecycle/spec.md#L47-L49)), and this is called out as a Windows file-locking risk in [design.md:65](openspec/changes/add-database-compatibility-epochs/design.md#L65). The tests cover successful reset, WAL/SHM cleanup, and future-version refusal, but none simulate a removal/recreation failure, so the wrapped-error branch at [database-lifecycle.ts:53-58](packages/store/src/database-lifecycle.ts#L53-L58) is unverified.
- **Recommendation:** Add a test that forces the reset to fail (e.g. a `DatabaseConstructor` stub that throws on the second construction, or a stubbed `removeDatabaseFiles`) and assert opening throws the `Failed to reset ... database compatibility epoch` error with the kind/epoch transition.
- **Fix:** Added a constructor-stub regression test that fails the post-removal reopen and asserts the focused reset error includes the database kind and epoch transition.
- **Verification:** Confirmed. [store.test.ts:187-222](packages/store/test/store.test.ts#L187-L222) creates an epoch-1 workspace db, then reopens at epoch 2 with a `FailingConstructor` that throws on its second construction (the post-removal reopen), asserting the error matches `Failed to reset workspace database compatibility epoch 1 -> 2: simulated reopen failure` â€” exercising the wrapped-error branch at [database-lifecycle.ts:76-81](packages/store/src/database-lifecycle.ts#L76-L81) and confirming the kind + epoch transition are surfaced and opening aborts before any store access. Suite green (20/20).

## Summary
Solid, well-scoped implementation that cleanly resolves RF5/RF6: the epoch is persisted independently of `PRAGMA user_version`, a single shared helper owns the fresh-vs-legacy-vs-mismatch decision, WAL/SHM companions are cleaned up, forward migrations are preserved within an epoch, and same-epoch downgrades fail safely. Tests, typecheck, and the developer-guidance doc are all present and the app-side `label` derivation is correct. All three review findings have been fixed and verified: RF1 (initial-open robustness â€” unreadable files now recover through the reset path with no handle leak), RF2 (future-version refusal is now provably write-free, enforced by a trigger-based test), and RF3 (the reset-failure branch is now covered). Final state: 20/20 store tests passing, `tsc --noEmit` clean. No open findings remain.
